\section*{Introduction}
Seperate chaining and open adressing are both collision management techniques used when implementing hash tables. Separate chaining relies on linked lists, initialized at each entry in the hash table, thus when a collision occurs it can be added to the linked list. Open addressing allows elements to be indexed to different addresses in the table when a collision occurs. Seperate chaining is a simpler technique to implement, which one could assume that the simplicity would indicate a more efficient implementation but it would turn out that, that is not the case.
\subsection*{Methodology}
To compare the two techniques, we will use experimental analysis. We will implement both techniques and then run them on the same data set. The following factors in the experimental analysis were considered and chose: 
\begin{itemize}
    \item \textbf{Programming Language}: C has been chosen as the language to implement the two schemes as the language has minimal abstraction overhead, and direct control of the memory layout since the two techniques can be heavily affected by the memory layout and allocation. The code was compiled using the \texttt{gcc} compiler.
    \item \textbf{Domain of experimentation}: Since we want to get an idea of the performance in all scenarios, all combinations of the following have been considered. \begin{itemize}
        \item \textbf{Load Factor}: Is tested from 0.5 to 0.95 in increments of 0.05
        \item \textbf{Map Size}: For this experiment fixed map sizes of $2^{n}$ where $n=\{5,6,7,...,24\}$ 
    \end{itemize}
    Additionally each of the following possible combinations was executed 10 times to get an average time for each combination.
    \item \textbf{Data selection:} The keys are generated using C's psuedo-random number generator using a fixed seed to ensure that the experiments are consistent for both techniques. 
    \item \textbf{Operations:} In an attempt to reproduce a realistic utilization of the hash maps, the following operations were performed in a fixed random order accross experiments with the following probabilities:
    \begin{itemize}
        \item 50\% of operations are insertions
        \item 40\% of operations are searches
        \item 10\% of operations are deletions
    \end{itemize}
    \item \textbf{Hashing function:} The following basic hashing function was used: \texttt{h(k) = k mod m} where $k$ is the key and $m$ is the size of the hash table. 
    \item \textbf{Timing:} Timing is performed at the nanosecond level using the \texttt{clock\_gettime} function with the \texttt{CLOCK\_MONOTONIC} clock, which is not affected by changes in the system time and provides high-resolution timing. Only the operations are timed to avoid timing overhead from the parts of the code running the experiment, a median is calculated from the 10 runs to get a accurate representation of the time to exclude outliers.
    \item \textbf{Hardware:} The experiments were run on a machine with the following specifications: \begin{itemize}
        \item CPU: Intel Core i7-1360P
        \item RAM: 32GB LPDDR5 (Dual Channel, 6000MHz)
        \item OS: Windows 11
    \end{itemize}
    In an attempt to reduce as much fluctuation in system performance, the device was put into airplane mode, with all other applications closed and the power settings were set to "Best Performance" to ensure that the CPU is running at its maximum performance level.
\end{itemize}

\subsection*{Conclusion}
The goal of this analysis was to compare the performance of seperate chaining and linear probing in a real worl application. The results of the experiments show that linear probing consistently outprerforms seperate chaining across almost all load factors. The most likely reasoning for this result is due to the memory layout of the two techniques, linear probing stores all of the elements in a continuous sequence in memory, which allows for better performance when using that hash tables, where as in seperate chaining each element in the hash table is a pointer to a linked list, which could be scattered anywhere in memory, which can lead to worse performance. The experiment unfortunately does have some holes that were not considered, for example resizing of maps was not considered. Additionally the use of adversarial inputs was not considered which may have demonstrated more significant performance differences. Additionally the use of a singular device for testing may have introduced bias, as well as the use of the windows operating system. Additionally the different operations could have been tested separately to get a better understanding but since we were representing the real world, the mixed operation analysis seemed more fitting. Although it is almost imposible to deterministically say that one algorithm is better then another, due to the wide variety of different uses and parameters. Based on what was experiment it is fair to conclude that on average, linear probing has better overall performance then separate chaining.
